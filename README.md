# Personal Knowledge Base

一个面向个人知识沉淀与回顾的知识库产品 demo。  
项目围绕“知识收集 -> 分类整理 -> 回顾复用 -> AI 提炼”这条主链路展开，重点展示我作为 AI 产品经理候选人对产品结构、MVP 范围、AI 能力落点以及 AI Coding 协作开发方式的理解。

## 项目简介

日常学习、阅读和工作中，用户会积累大量零散信息，但常见问题是：

- 知识记录方式分散，难以统一管理
- 存下来的内容缺少分类和标签，后续检索效率低
- 很多知识“只存不看”，没有形成持续回顾机制
- 长文本不易快速提炼，知识复用成本高

Personal Knowledge Base 的目标，是让用户把知识作为“可整理、可检索、可回顾、可提炼”的资产来管理。

## 核心功能

- 知识收集：新增知识条目，记录标题、正文、来源、分类、标签等信息
- 分类整理：支持分类和标签管理，提升后续检索效率
- 搜索筛选：支持关键词搜索、分类筛选、标签筛选
- 收藏归档：支持收藏重要知识、归档已处理条目
- 知识详情：支持查看完整内容、来源、标签、元信息、AI 总结区
- 知识回顾：提供“今日回顾 / 最近回顾”入口，帮助用户重新看到旧知识
- AI 自动摘要：为知识生成摘要、建议分类、建议标签和关键点

## 当前 Demo 实现

当前仓库是一个前端可运行 demo，重点验证交互与产品结构。

- 前端：React + Vite + Tailwind CSS
- 路由：React Router
- 数据存储：浏览器 LocalStorage
- UI：自定义组件 + Radix UI 组件库
- 部署：GitHub Pages

## 环境变量

当前 demo 前端版本不强依赖环境变量。

```env

OPENAI_BASE_URL="https://your-openai-compatible-endpoint"
OPENAI_API_KEY="your_api_key"

```

## 数据库初始化

当前 demo 使用 LocalStorage，因此无需数据库初始化。

如果扩展到 Prisma + SQLite 版本，初始化流程如下：

```bash
npx prisma generate
npx prisma migrate dev --name init
```

## 接口说明

当前仓库为前端演示版，暂未实际接入后端接口。

完整版接口规划包括：

- `POST /api/knowledge`
  - 新增知识条目
- `GET /api/knowledge`
  - 查询知识列表，支持搜索与筛选
- `PATCH /api/knowledge/[id]`
  - 更新知识条目
- `DELETE /api/knowledge/[id]`
  - 删除知识条目
- `POST /api/knowledge/ai-summary`
  - 生成 AI 摘要、建议分类、建议标签和关键点

推荐统一返回结构：

```ts
type ApiSuccess<T> = {
  success: true;
  data: T;
  message?: string;
};

type ApiError = {
  success: false;
  error: string;
  message: string;
};
```

## 面试讲稿（3 到 5 分钟）

下面这段内容可以直接作为面试时的项目介绍：

### 1. 为什么做这个产品

这个项目叫 Personal Knowledge Base，我想解决的是一个很典型但经常被忽略的问题，就是很多人每天都在记录知识，但这些知识其实没有真正沉淀成资产。用户可能在备忘录、文档、网页收藏夹里存了很多内容，但后面很难系统整理，也很难再回头看。

### 2. 解决了什么问题

我希望这个产品解决三个核心问题。  
第一，知识收集分散，缺少统一入口。  
第二，知识存下来了但缺少分类、标签和筛选，后面很难找。  
第三，也是我觉得最重要的一点，就是知识通常“只存不看”，所以我在这个项目里专门加入了“知识回顾”这个能力。

### 3. 目标用户是谁

目标用户主要是有持续输入和记录习惯的人，比如学生、知识工作者、内容从业者，或者像我这种会长期积累学习笔记、产品思考和阅读记录的人。

### 4. MVP 怎么定义

这个项目的 MVP 我定义得比较克制，重点不是做一个很重的知识管理系统，而是先验证一条最核心的闭环：  
用户能不能方便地记录知识，能不能把知识整理好，后面能不能再回来看，并且能不能借助 AI 快速提炼重点。

所以 MVP 只保留了最关键的功能，包括：

- 新增知识
- 分类和标签管理
- 搜索和筛选
- 收藏和归档
- 知识详情页
- 今日回顾 / 最近回顾
- AI 自动摘要

### 5. 核心页面和功能是什么

首页主要是知识库的总览，会展示统计信息、快速新增入口、最近添加和最近回顾。  
知识详情页主要承载完整内容查看、来源展示、标签、收藏归档和 AI 总结区。  
另外我还做了一个回顾页面，用来集中展示最近添加、最近未查看、以及用户标记了稍后回顾的知识，这个页面是我觉得比较能体现产品价值的地方。

### 6. AI 放在哪个环节

我没有把 AI 放成一个很虚的入口，而是放在最合适的整理环节。  
比如用户新增一条知识之后，AI 可以自动生成摘要、建议分类、建议标签和关键点。这样用户不需要自己从长文本里再做一次手工提炼，能大幅降低整理成本。  
从 MVP 角度，我认为最值得优先接入的两个 AI 能力就是自动摘要，以及自动标签和自动分类。

### 7. 我如何使用 AI Coding 工具辅助开发

在开发这个项目时，我把 AI Coding 工具主要当成三个角色来用。  
第一是快速搭建前端页面和交互原型，帮助我提升实现速度。  
第二是辅助我补齐知识详情页、回顾页、README、异常处理和测试用例这些结构化内容。  
第三是帮助我把产品思路快速转化为可以演示的代码和文档。  
我自己重点负责的是产品结构、功能优先级判断、AI 能力设计和 demo 取舍，而不是完全把开发交给工具。

### 8. 项目亮点

我觉得这个项目有几个亮点。  
第一，它不是单纯的 CRUD 页面，而是把“知识回顾”加入进来，让知识管理更接近真实用户价值。  
第二，AI 能力放得比较克制，是围绕整理环节增强，而不是为了加 AI 而加 AI。  
第三，这个项目虽然是一个 demo，但已经具备从前端原型演进到 Next.js + Prisma + SQLite + OpenAI 兼容接口全栈版本的清晰路径。

### 9. 后续迭代方向

如果继续做，我下一步会补三块。  
第一，接入真正的后端和数据库，实现跨设备同步。  
第二，把 AI 自动摘要能力真正接入接口。  
第三，把回顾能力做得更完整，比如增加“标记已回顾”、回顾提醒和更细粒度的知识再激活机制。

整体来说，这个项目是我把产品思考、MVP 取舍、AI 能力设计和 AI Coding 协作方式结合起来的一次实践。
