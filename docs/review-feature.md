# 回顾功能设计

## 这个功能解决什么问题

个人知识库的核心问题不是“记不下来”，而是“存进去以后不再回看”。  
回顾功能解决的是知识沉淀后缺乏再次激活的问题，让用户能重新看到旧知识、重新理解上下文、提升知识复用率。

## 用户什么时候使用

1. 每天打开应用时，先看“今日回顾”
2. 学完一条内容后，给它安排一个稍后回顾时间
3. 想知道哪些知识很久没看过时，进入回顾页统一查看

## 最适合 MVP 的交互方式

不做复杂记忆算法，直接采用三个简单来源：

1. 最近添加的知识
2. 最近未查看的知识
3. 用户手动标记了回顾时间的知识

这样最轻量，也最容易在面试里解释清楚。

## 页面应该怎么呈现

页面名建议为：
- 今日回顾
或
- 最近回顾

页面结构建议：

1. 顶部说明区
- 告诉用户今天有多少条知识值得回顾
- 强调“知识回顾”是产品价值，而不是单纯存储

2. 今日待回顾
- `reviewAt` 正好在今天的条目

3. 最近添加
- 最近新创建的知识，适合作为第一次复看入口

4. 最近未查看
- `lastReviewedAt` 为空，或者距离当前时间较久的条目

5. 稍后回顾
- 用户手动安排了未来回顾时间的条目

## 最轻量的实现方案

1. 继续使用前端本地存储
2. 直接复用已有字段：
- `createdAt`
- `updatedAt`
- `reviewAt`
- `lastReviewedAt`

3. 如果没有 `lastReviewedAt`，就视为“未查看”
4. 后续详情页可补一个“标记为已回顾”按钮，写入 `lastReviewedAt`

## 为什么适合面试 demo

1. 一眼能看懂
2. 能体现“知识需要回顾”的产品思考
3. 不依赖后端和复杂算法，演示成本低
4. 很适合后续再叠加 AI，比如“AI 推荐今日最值得回顾的知识”
